Governance

Three-tier access with 2/3 supermajority rules

Dakota governance separates structural authority, day-to-day operations, and root membership while preserving deterministic quorum logic.

Tier 1

Root Overlords

Read dynamically from the validator contract and can add or remove non-root overlords directly.

Tier 2

Overlords

Vote on structural network and contract changes under the 2/3 supermajority rule.

Tier 3

Guardians

Handle operational upgrade triggers through ProxyAdmin and establish one-time proxy linkage.

Governance rules

  • Supermajority quorum: (totalVoterCount * 2 + 2) / 3
  • Vote tallies expire after a configurable threshold, defaulting to 1,000 blocks
  • Voter pool changes are blocked while any tally remains active
  • Permanent management revocation is possible in ordered phases
  • Each core contract maintains an independent voter pool with external contract support
Quorum formula 2/3 supermajority

Governance is tuned for federation without sacrificing explicit consensus thresholds.

Services

Ecosystem services overview

Dakota.Cards is where Cryft Labs can surface original innovations as ecosystem services. The portal is built to let new experiences sit on the same Dakota infrastructure while reusing shared governance, privacy, and redeemable-code building blocks, including KotaTM, the AI-assisted gifting widget named as a shorthand for Dakota.

Cryft Labs service design

  • Dakota.Cards provides the ecosystem surface for original Cryft Labs innovations rather than acting as a single-product website
  • KotaTM is the Cryft Labs AI-assisted embeddable gifting widget, named as a shorthand for Dakota, and helps clients implement Dakota Network and the proprietary redeemable-code system more easily
  • Each service can inherit the same account model, governance posture, contract routing, and privacy architecture without rebuilding the full stack
  • Future management tools are planned to extend administration, lifecycle control, and service operations across the ecosystem

Paladin privacy and redeemable-code invention

  • Paladin with the Pente privacy domain allows service flows to keep sensitive redemption data private while still coordinating with transparent onchain state
  • Cryft Labs uses a two-contract invention pair: CodeManager for public routing and lifecycle control, and PrivateComboStorage for private validation data inside the privacy domain
  • Together they create transparent but secure management of redeemable codes, so users and operators can trust the flow without exposing the private combination material
  • That pattern can be reused by additional Dakota services that need controlled redemption, privacy-aware issuance, or secure claim workflows
